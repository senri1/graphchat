The token thing was me naming the bad path so we avoid it. You are describing the good path.

**How I’d Do It**
Keep GraphChat’s UI and graph behavior mostly unchanged, but add a second OpenAI runtime:

`OpenAI API key mode`
Current path: [src/services/openaiService.ts](/Users/sen/Code/graphchatv1/src/services/openaiService.ts) calls `/v1/responses`.

`OpenAI via Codex mode`
New path: Electron spawns `codex app-server`, signs in with ChatGPT, then uses `thread/start` and `turn/start`.

For minimal behavior change, I would **not** make Codex own GraphChat’s conversation graph at first. I’d keep GraphChat as the source of truth and create a fresh Codex thread per generation:

1. Build the same graph-context request using `buildOpenAIResponseRequest` in [src/llm/openai.ts](/Users/sen/Code/graphchatv1/src/llm/openai.ts).
2. Adapt that into Codex input:
   - prior messages: inject as history with `thread/inject_items`, or serialize them into a transcript for v1
   - current user node: send to `turn/start`
   - selected model: pass as `model`
   - reasoning effort: pass as `effort`
3. Stream `item/agentMessage/delta` into the assistant node, the same way your app currently streams OpenAI deltas.
4. Cancel with `turn/interrupt`.

That preserves GraphChat’s branching model much better than trying to reuse one persistent Codex thread forever.

**Important Caveats**
This can be very close to current behavior for normal chat, streaming, model selection, cancellation, and reasoning summaries.

The parts that will not be identical on day one are GraphChat’s current OpenAI function tools, LaTeX/graph tool loop, API payload viewer, and possibly web search. Codex has its own tool/event system, and dynamic tools exist but are experimental. I’d ship basic text/image chat first, then map the tool loop after the transport is solid.

Also, this is desktop-only unless you run a local companion service, because the browser cannot spawn `codex app-server`.

Docs backing this: [Codex App Server](https://developers.openai.com/codex/app-server), especially `account/login/start`, `account/rateLimits/read`, `model/list`, `thread/start`, `thread/inject_items`, `turn/start`, and streamed `item/agentMessage/delta` events.
