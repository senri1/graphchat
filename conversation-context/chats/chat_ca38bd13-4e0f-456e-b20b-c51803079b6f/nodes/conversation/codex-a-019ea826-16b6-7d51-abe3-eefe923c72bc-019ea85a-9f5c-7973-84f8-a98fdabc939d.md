Yes, exactly, but **read-only initially**.

The milestone would be:

1. Select an existing Codex project folder.
2. Use `thread/list` filtered by that folder’s `cwd`.
3. Load each thread and its turns.
4. Convert prompts and responses into GraphChat nodes.
5. Connect native forks using `forkedFromId`.
6. Display unrelated threads as separate graph roots.
7. Show reasoning summaries, commands, tool activity, and file changes in node details.

This lets us validate the mapping between real Codex data and the graph UI without risking thread corruption.

After that works, the next milestone adds mutation:

- Continue an existing thread from its tip.
- Branch from an older node using fork and rollback.
- Stream new responses into graph nodes.
- Open the resulting thread in either GraphChat or the official Codex app.

One caveat: existing Codex forks tell us their parent thread, but may not identify the exact historical turn where the fork occurred. We can reconstruct shared prefixes where the histories match, but GraphChat-created forks will be exact because we store the branch-point `turnId`.

So yes: **first make GraphChat an excellent viewer for the Codex projects and threads you already have; then make it a fully interoperable editor.**
