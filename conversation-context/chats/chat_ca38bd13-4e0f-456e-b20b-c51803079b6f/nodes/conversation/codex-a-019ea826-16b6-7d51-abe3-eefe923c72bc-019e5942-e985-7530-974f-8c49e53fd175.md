The “App Server” is a **local Codex runtime process** your app talks to. It is not your backend and not the normal OpenAI API. You spawn it on the user’s machine, usually with:

```bash
codex app-server
```

Then GraphChat talks to it over JSON-RPC. The app-server talks to OpenAI/Codex using the user’s ChatGPT login, manages Codex threads, streams events back, handles tools/approvals, and exposes things like model listing and rate-limit/account state. Official docs describe it as a JSON-RPC server with methods like `account/login/start`, `model/list`, `thread/start`, `thread/fork`, and `turn/start` ([Codex App Server docs](https://developers.openai.com/codex/app-server)).

**Basic API Flow**
With the normal Responses API, your app does this:

```text
GraphChat
  -> OpenAI HTTPS API /v1/responses
      model: gpt-...
      input: messages/history
      tools: tool definitions
      api key: sk-...
  <- response / stream events
```

Your app owns almost everything:
- API key storage
- billing through API account
- model registry
- conversation history
- branching
- tool execution loop
- retries/cancel/payload storage

The model may ask for a tool call, but your app executes the tool and sends the tool result back. OpenAI docs describe this style as request-based; conversation state is either manually sent each turn, chained with `previous_response_id`, or stored via Conversations ([conversation state docs](https://developers.openai.com/api/docs/guides/conversation-state)). Tools are usually configured in the `tools` parameter ([tools docs](https://developers.openai.com/api/docs/guides/tools)).

**Codex App Server Flow**
With Codex App Server, your app does this:

```text
GraphChat
  -> local codex app-server JSON-RPC
      account/login/start
      model/list
      thread/start or thread/fork
      turn/start
  -> Codex/OpenAI backend using ChatGPT auth
  <- streamed Codex events
```

Now the app-server owns more of the agent harness:
- ChatGPT/Codex login
- ChatGPT plan/rate-limit awareness
- Codex thread history
- model list from Codex
- turn streaming
- tool progress
- approvals
- sandbox/tool policy
- thread fork/resume/rollback/compact

So instead of “send one HTTPS request and get one response,” you say: “start or resume this Codex thread, begin a turn, stream me the events.”

**The Key Difference**
The API is a **model endpoint**.

Codex App Server is an **agent runtime**.

That’s the main mental shift. The API gives you primitives to build your own chat/agent system. Codex App Server gives you a local control surface for the Codex agent system.

**Comparison**
| Concern | Responses API | Codex App Server |
|---|---|---|
| Auth | API key | ChatGPT login or API key |
| Usage limits | API billing/limits | ChatGPT/Codex plan limits when using ChatGPT auth |
| Transport | HTTPS/WebSocket API | Local JSON-RPC to `codex app-server` |
| Unit of work | Response | Turn inside a thread |
| Conversation state | Your app, `previous_response_id`, or Conversations API | Codex thread |
| Branching | Your app builds it | `thread/fork` exists natively |
| Tools | Your app defines/executes tool loop | Codex harness manages tools, events, approvals; client participates |
| Streaming | Response stream events | Turn/item notifications like agent message deltas |
| Best for | General custom apps | Custom Codex clients |

For GraphChat, the clean version is: GraphChat remains the visual UI and storage layer, but Codex App Server becomes the OpenAI/Codex runtime. So selecting an OpenAI model in GraphChat would eventually mean:

```text
GraphChat node branch
  -> find/fork Codex thread for that branch
  -> turn/start with the user input
  -> stream assistant text into GraphChat node
  -> store codexThreadId / turnId on that node
```

That gets you the thing you want: your own UI, but using ChatGPT/Codex auth and Codex’s native runtime instead of the normal API-key Responses path.
