Yes. The folder button currently selects the **local project directory in which Codex will operate**. “Workspace selected” would be clearer as **“Codex working folder selected.”**

For example, selecting:

```text
/Users/sen/Code/my-project
```

means future Codex turns in that GraphChat chat use that path as their `cwd`.

**What Selecting It Does**

1. Opens the operating system’s folder picker.
2. Saves the selected path in the active GraphChat chat’s metadata.
3. Highlights the folder button.
4. Passes that path to Codex whenever GraphChat starts, resumes, or forks a thread.
5. Records the path in completed nodes’ Codex references.

The association survives restarting GraphChat.

**What Codex Does With It**

The folder establishes Codex’s project context:

- Commands run relative to that directory.
- File tools inspect files inside that project.
- File edits are scoped to that workspace.
- Codex discovers relevant `.git`, `AGENTS.md`, `.codex/config.toml`, skills, rules, and project configuration from that location.
- Threads created through GraphChat carry that `cwd`, allowing the official Codex app to recognize their project context.

Our current thread configuration is:

```ts
{
  cwd: selectedFolder,
  sandbox: "workspace-write",
  approvalPolicy: "on-request"
}
```

So Codex may modify files within that workspace. When the Codex policy requires permission for a command or file operation, GraphChat receives and displays an approval request.

**How It Affects Conversations**

Suppose the selected folder is `project-a`.

A new root conversation does approximately:

```text
thread/start(cwd = project-a)
turn/start(...)
```

Continuing from the latest node does:

```text
thread/resume(existingThread, cwd = project-a)
turn/start(...)
```

Branching from an older node does:

```text
thread/fork(existingThread, cwd = project-a)
thread/rollback(...)
turn/start(...)
```

Each completed assistant node stores something like:

```ts
{
  threadId: "...",
  turnId: "...",
  cwd: "/Users/sen/Code/project-a",
  model: "gpt-5.4",
  effort: "high"
}
```

**What It Does Not Do**

Selecting a folder does not:

- Upload the folder to GraphChat or Google Drive.
- Copy files into GraphChat storage.
- Change where GraphChat stores its graphs.
- Import existing Codex threads from that project.
- Create a Git worktree.
- Add the project to the official Codex app’s sidebar directly.
- Give GraphChat ownership of the project.

It simply associates the GraphChat chat with a local Codex working directory.

**One Design Issue**

Currently, the folder is stored once per GraphChat chat. You can change it later, and future continuations and forks will use the new folder even if the original Codex thread was created in the old folder.

That is too implicit for our interoperability goal. I think the intended rule should be:

- A GraphChat conversation graph is associated with one Codex project folder.
- Existing branches retain that project association.
- Changing it should be an explicit “move/rebind project” operation.
- GraphChat should warn if a node’s stored `cwd` differs from the currently selected project.
- Worktree branches may have separate directories, but those should be modeled explicitly.

I would also rename the UI from **Workspace** to **Project folder** or **Codex working folder**, because “workspace” can also mean a ChatGPT organization/workspace. The current button represents a filesystem directory, not your ChatGPT account workspace.

OpenAI describes the Codex app project model similarly: a project corresponds roughly to starting a Codex session in a specific directory, and Codex scopes tools to that project. [Codex app features](https://developers.openai.com/codex/app/features/)
