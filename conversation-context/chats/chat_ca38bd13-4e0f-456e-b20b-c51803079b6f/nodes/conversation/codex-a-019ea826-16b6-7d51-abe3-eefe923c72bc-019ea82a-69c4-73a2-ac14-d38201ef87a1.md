## Three Layers

### 1. GraphChat graph

GraphChat owns the visible structure:

```text
A ‚Üí B ‚Üí C
    ‚îî‚Üí F
```

It stores node positions, edges, titles, text, attachments, and which Codex checkpoint produced each response.

### 2. Codex threads

Each Codex thread is essentially a durable, linear conversation history:

```text
Thread T1: A ‚Üí B ‚Üí C
Thread T2: A ‚Üí B ‚Üí F
```

Codex stores both histories internally. GraphChat continues showing shared nodes `A` and `B` only once.

### 3. Codex turns

A turn is one unit of agent work:

```text
User input
  ‚Üí reasoning
  ‚Üí tool calls
  ‚Üí approvals
  ‚Üí tool results
  ‚Üí final assistant response
```

All of that belongs to one turn.

## Node Metadata

For `A ‚Üí B ‚Üí C`, GraphChat might store:

```text
A: threadId=T1, turnId=TA, turnOrdinal=1
B: threadId=T1, turnId=TB, turnOrdinal=2
C: threadId=T1, turnId=TC, turnOrdinal=3
```

`turnId` and `turnOrdinal` are different:

- `turnId` is Codex‚Äôs opaque identifier, such as `turn_abc123`.
- `turnOrdinal` is GraphChat‚Äôs checkpoint position within the thread.

We do not really need both ‚Äúturn number‚Äù and ‚Äúcompleted turn count.‚Äù Those represent roughly the same checkpoint. The authoritative reference should be `turnId`; the ordinal is useful for calculating rollbacks.

## Branching From B

Suppose `T1` currently contains:

```text
T1: A ‚Üí B ‚Üí C
```

You select `B` and send prompt `F`.

### Step 1: Fork

GraphChat calls:

```text
thread/fork(T1)
```

Codex creates a new thread:

```text
T2: A ‚Üí B ‚Üí C
```

At this instant, `T2` is a complete copy of `T1`. Codex records that `T2.forkedFromId = T1`.

### Step 2: Roll Back

GraphChat wants the state at `B`, so it calls:

```text
thread/rollback(T2, numTurns=1)
```

Now:

```text
T1: A ‚Üí B ‚Üí C
T2: A ‚Üí B
```

The original `T1` is untouched.

### Step 3: Start F

GraphChat calls:

```text
turn/start(T2, input=F)
```

The result becomes:

```text
T1: A ‚Üí B ‚Üí C
T2: A ‚Üí B ‚Üí F
```

GraphChat renders:

```text
A ‚Üí B ‚Üí C
    ‚îî‚Üí F
```

`F` receives metadata such as:

```ts
{
  threadId: "T2",
  turnId: "TF",
  turnOrdinal: 3,
  forkedAtNodeId: "B"
}
```

## Is Codex Itself a Graph?

Not in the same sense as GraphChat.

Codex primarily has:

- Linear histories inside threads
- Threads that may be related through `forkedFromId`
- Turns and items inside each thread

That gives Codex a **tree of linear threads**, but not GraphChat‚Äôs arbitrary visual node graph.

Also, `forkedFromId` tells us that `T2` came from `T1`, but it does not fully express the visible branch point `B`. GraphChat must retain that mapping.

Therefore:

> GraphChat owns the graph. Codex owns the executable conversation histories.

## The Operations

### `thread/start`

Creates a completely new Codex conversation:

```text
nothing ‚Üí Thread T1
```

It establishes thread-level configuration such as:

- Working directory
- Model
- Sandbox
- Approval policy
- Developer instructions
- Tool configuration

It does not itself send a user message.

### `turn/start`

Adds input to an existing thread and starts agent work:

```json
{
  "threadId": "T1",
  "input": [
    { "type": "text", "text": "Explain this code" }
  ]
}
```

The request immediately returns a new turn object. App Server then streams events such as:

```text
turn/started
item/started
item/agentMessage/delta
item/completed
turn/completed
```

One turn can contain many items: user messages, reasoning summaries, commands, file edits, approvals, and the final response.

### `thread/fork`

Creates a new thread by copying an existing thread‚Äôs stored history:

```text
T1: A ‚Üí B ‚Üí C

thread/fork(T1)

T2: A ‚Üí B ‚Üí C
```

There is currently no `atTurnId` parameter. It forks the thread at its present endpoint. That is why branching from an older GraphChat node requires a rollback afterward.

### `thread/rollback`

Removes the last `N` turns from one thread‚Äôs model context:

```text
T2: A ‚Üí B ‚Üí C

thread/rollback(T2, 1)

T2: A ‚Üí B
```

Important: rollback changes conversation history, but **does not undo filesystem changes** previously made by Codex tools. If turn `C` modified files, GraphChat or Git must handle reverting those files separately.

### `thread/resume`

After restarting GraphChat or App Server:

```text
thread/resume(T2)
turn/start(T2, ...)
```

This reloads the persisted history so the conversation can continue.

## Correctness Detail

When branching, we should not blindly trust a stored count. We should:

1. Call `thread/read` with turns included.
2. Find `B` using its stored `turnId`.
3. Count how many turns follow it.
4. Fork the thread.
5. Roll back exactly that many turns.
6. Verify the fork now ends at `B`.
7. Start the new turn.

That protects us against interrupted turns, compaction, retries, and other history changes.
