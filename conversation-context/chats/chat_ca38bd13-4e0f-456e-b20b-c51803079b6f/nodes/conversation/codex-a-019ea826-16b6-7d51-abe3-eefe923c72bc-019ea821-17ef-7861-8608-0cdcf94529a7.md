Next, I’d build the **real GraphChat-to-Codex conversation path**, including native branching.

The important discovery: `thread/fork` copies the current thread, not an arbitrary historical turn. So each graph node should retain a Codex checkpoint:

```ts
{
  threadId,
  turnId,
  completedTurnCount
}
```

Then:

- Continuing from the branch tip uses `turn/start` on the existing thread.
- Branching from an older node uses `thread/fork`, followed by `thread/rollback` to that node’s turn count.
- The new prompt starts on the fork.
- The original branch remains unchanged.

**Milestone scope**

1. Replace the composer’s generation call with a `CodexRuntime` adapter.
2. Stream Codex output directly into a new graph node.
3. Persist thread and turn mappings alongside graph nodes.
4. Resume threads after restarting the app.
5. Support cancellation, errors, retry, model, and reasoning effort.
6. Associate each chat with a workspace folder and pass it as Codex’s `cwd`.
7. Route native tool approvals through the GraphChat UI.
8. Preserve portable node text so the graph remains viewable if its local Codex thread is unavailable.

**Acceptance test**

We should be able to:

1. Ask question A.
2. Continue with question B.
3. Return to A and create an alternative B.
4. See that the two branches use different Codex threads with the correct shared history.
5. Restart GraphChat and continue either branch successfully.
6. Run a tool in the selected workspace with an approval prompt.

That gets the hardest architectural part right. Afterward, tools, richer Codex events, syncing, and mobile behavior can build on something real instead of a temporary compatibility layer.
