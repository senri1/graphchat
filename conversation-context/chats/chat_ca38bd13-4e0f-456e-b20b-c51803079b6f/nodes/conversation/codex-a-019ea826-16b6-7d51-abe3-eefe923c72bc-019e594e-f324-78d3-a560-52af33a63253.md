Yes. You’re basically picturing it correctly: **Codex App Server is a local program with local state**, and your UI talks to that program instead of talking directly to `/v1/responses`.

A good mental model:

```text
GraphChat UI
  talks JSON-RPC to
Codex app-server process
  which owns auth, threads, tools, sandbox, local execution, streaming
  and talks to
OpenAI/Codex backend using your ChatGPT or API-key auth
```

**What Data Does It Have?**
There are a few buckets:

1. **Account/auth data**
   Codex can store API-key auth or managed ChatGPT auth. In ChatGPT mode, Codex owns the OAuth/device-code flow, persists tokens, and refreshes them. The app-server exposes `account/read`, `account/login/start`, `account/logout`, and `account/rateLimits/read` for this. Source: [Codex App Server auth endpoints](https://developers.openai.com/codex/app-server).

2. **Thread/conversation data**
   Yes: threads are part of Codex’s local data. A thread is a conversation between the user and the Codex agent. A turn is one user request plus the agent work that follows. Items are messages, command runs, file changes, tool calls, and so on. The docs say thread logs are persisted as JSONL files on disk, and app-server can list/read/resume/fork/archive them. Source: [Codex App Server threads](https://developers.openai.com/codex/app-server).

3. **Config data**
   Codex has user config at `~/.codex/config.toml`, plus optional project `.codex/config.toml` in trusted projects. This config controls model/provider defaults, approval policy, sandbox settings, MCP servers, instructions, project docs, etc. Source: [Codex configuration reference](https://developers.openai.com/codex/config-reference).

4. **Project/workspace data**
   Each thread can be associated with a `cwd`, meaning “this is the folder Codex is working in.” In the Codex app, that’s why a thread feels attached to a project folder. When you start a thread via app-server, `thread/start` can include `cwd`, `sandbox`, `approvalPolicy`, model, personality, and more. Source: [Codex App Server start thread](https://developers.openai.com/codex/app-server).

5. **Worktree data**
   In the Codex app, worktrees are Codex-managed Git worktrees under `$CODEX_HOME/worktrees`. Those let Codex work in parallel without disturbing your main checkout. Source: [Codex worktrees](https://developers.openai.com/codex/app/worktrees).

**Does It Store A List Of Tools?**
Partly.

The **built-in Codex tools** are more like part of the Codex harness: shell execution, file operations, apply-patch style editing, web search, image generation, review mode, browser/computer-use features where available, etc. They are not just a user-editable list sitting in a JSON file. They are capabilities of the Codex runtime.

But the **enabled external/custom tools** are definitely configuration/data-driven:
- MCP servers live in Codex config.
- Plugins can contribute MCP servers.
- Tool approval modes can be configured.
- Sandbox/permission profiles decide what local command/file/network tools can actually do.

The docs say the Codex app, CLI, and IDE extension share MCP settings, and MCP config lives in `config.toml` or project `.codex/config.toml`. Source: [Codex MCP](https://developers.openai.com/codex/mcp).

**What Is The “Harness”?**
The harness is the orchestration layer around the model.

The plain API is mostly:

```text
send input + tools -> get response/tool calls -> you handle the rest
```

The Codex harness is:

```text
manage thread
assemble context
choose model/provider
apply instructions/personality/project docs
expose tools
run commands/filesystem tools
enforce sandbox and approvals
stream events
persist turns/items
handle fork/resume/rollback/compact
handle auth/rate limits
```

So the harness is not just “the model.” It is the local agent runtime that makes the model behave like Codex.

**Folder/Project Association**
Yes, that is app-server related. A Codex thread can have a `cwd`. That working directory affects what files the tools can see, what project docs/config are loaded, what sandbox roots apply, what thread filters show in `thread/list`, and where shell/file tools operate.

So in GraphChat terms, a node/branch could store:

```ts
{
  codexThreadId: "thr_...",
  codexCwd: "/Users/sen/Code/graphchatv1",
  model: "gpt-5.4",
  sandbox: "workspaceWrite"
}
```

Then continuing that branch means:

```text
thread/resume or thread/fork
turn/start with same or changed cwd/model/settings
stream item/agentMessage/delta back into GraphChat
```

**Compared To Basic API**
The API is like calling a model function.

The app-server is like talking to a local Codex engine that already knows how to run a full coding-agent session.

For your goal, GraphChat would not merely “send a request to OpenAI.” It would become a **Codex client**: it logs in through app-server, asks app-server for models/rate limits, starts/forks/resumes threads, and renders the streamed thread events in your graph UI.
