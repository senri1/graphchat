I blurred two different things:

- **Internal thread hydration:** being able to call `thread/read` and understand persisted Codex turns.
- **User-facing importer:** loading all existing Codex projects and conversations into GraphChat.

We need the first early. We do **not** need to build the second first.

I agree with your original instinct: start with basic GraphChat conversation and branching.

## Better Order

1. Choose a project folder as the Codex `cwd`.
2. Start a new Codex thread from GraphChat.
3. Send `A`, then continue with `B`.
4. Stream responses into GraphChat nodes.
5. Store each node’s `threadId` and `turnId`.
6. Branch from an older completed node using:
   - `thread/read` to locate the exact turn
   - `thread/fork`
   - `thread/rollback`
   - `turn/start`
7. Restart GraphChat and restore the graph/thread mappings.
8. Confirm both resulting threads appear in the official Codex app.
9. Only afterward, add importing arbitrary existing Codex threads and project forests.

The first acceptance graph should be:

```text
A → B → C
    └→ D
```

Internally:

```text
Thread T1: A → B → C
Thread T2: A → B → D
```

So `thread/read` is part of the conversation runtime and recovery machinery, not an importer milestone.

Once GraphChat-created threads interoperate correctly with the official Codex app, importing older Codex conversations becomes a straightforward extension of the same adapter. That is the path I’d take.
