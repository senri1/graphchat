I would not complete either large milestone in isolation. I’d build a **narrow interoperable vertical slice**, with import first because mutation depends on knowing how persisted Codex threads map into our graph.

## Recommended Order

1. Select a project folder and save its `cwd`.
2. Use `thread/list` and `thread/read`.
3. Render one existing Codex thread as GraphChat nodes.
4. Store GraphChat layout separately from Codex history.
5. Continue that thread from its tip using `turn/start`.
6. Branch from an older node using `thread/fork` and `thread/rollback`.
7. Restart both apps and verify the same conversations remain accessible.
8. Expand from one thread to the complete project forest.

So we are not spending weeks making a polished read-only importer. We build the minimum importer needed, then immediately add native generation and branching.

## Why Import Comes First

Starting with GraphChat-created conversations alone could lead us to design storage that works only for our own threads.

Importing first forces us to establish correctly:

- Project identity through `cwd`
- Thread and turn identity
- Conversion of Codex turns into visible nodes
- Separation between Codex history and GraphChat layout
- Behavior when GraphChat metadata is absent
- Compatibility with official Codex-created threads
- Reloading and reconciliation after either app changes a thread

Those are foundations the write path needs anyway.

## First End-to-End Test

Start with an existing Codex thread:

```text
Codex thread T1: A → B
```

GraphChat imports it:

```text
A → B
```

Then GraphChat continues it:

```text
A → B → C
```

The official Codex app should now show `C` in the same thread.

Then branch from `B`:

```text
A → B → C
    └→ D
```

Internally:

```text
T1: A → B → C
T2: A → B → D
```

Finally, reopen the official Codex app and confirm both threads are visible under the same project.

## Initial Scope

For the first implementation, I would support:

- One selected project
- Existing non-archived threads
- Text prompts and agent responses
- One imported thread rendered at a time
- Continue from tip
- Branch from a completed turn
- Streaming
- Stop, errors, model and reasoning effort
- Persistence and restart recovery

Then we expand to:

- All project threads in one forest
- Commands and file-change details
- Approvals
- Reasoning summaries
- Archived threads
- Concurrent updates from the official app
- Inferred branch points for older forks

So my recommendation is: **begin with import as the hydration foundation, but make the actual milestone import + continue + branch.** That proves interoperability rather than producing merely a viewer or an isolated GraphChat runtime.
