Yes: **GraphChat should store the visible node data itself.**

## What GraphChat Stores

For every visible node:

```ts
type GraphNode = {
  id: string;
  parentId: string | null;

  // GraphChat-owned presentation data
  text: string;
  title?: string;
  position: { x: number; y: number };
  attachments: Attachment[];
  createdAt: number;

  // Pointer into Codex history
  codex?: {
    threadId: string;
    turnId: string;
  };
};
```

GraphChat owns:

- Node text
- Graph edges
- Positions and dimensions
- Attachments
- Titles and visual settings
- Which Codex turn produced the node

Codex also retains the messages internally, but GraphChat keeps its own copy so the graph can render without loading every Codex thread and can still be exported or synchronized independently.

## “Codex Checkpoint”

“Codex checkpoint” was terminology I introduced. It is **not an official Codex object**.

A better name is:

```ts
type CodexConversationRef = {
  threadId: string;
  turnId: string;
};
```

It means:

> This GraphChat node represents the conversation state immediately after this Codex turn.

The `turnId` identifies the exact turn. We can derive its position by reading the thread. A stored turn number would only be a cache.

## What a Codex Thread Contains

A Codex thread is absolutely **not** a lightweight list of GraphChat node IDs.

Codex knows nothing about GraphChat node IDs unless we deliberately send/store one as client metadata.

A thread contains a durable event history resembling:

```text
Thread metadata
├── working directory
├── model and provider
├── sandbox and approval configuration
├── developer instructions
├── Turn 1
│   ├── user message
│   ├── reasoning data
│   ├── tool call
│   ├── tool result
│   └── assistant message
├── Turn 2
│   ├── user message
│   ├── file edit
│   └── assistant message
└── compaction/history records
```

Locally, I confirmed that your Codex installation persists records including:

- User and assistant messages
- Encrypted reasoning content
- Reasoning summaries
- Tool calls and outputs
- Turn configuration
- Model and effort
- Working directory
- Approval and sandbox configuration
- Fork relationships
- Token and rate-limit events

So the mapping is:

```text
GraphChat node
    └── points to Codex thread + turn

Codex turn
    └── contains the actual agent interaction that produced that node
```

## Two Copies of Text

For a generated response, the text exists in both places:

```text
GraphChat storage:
    Node.text = "The answer is..."

Codex storage:
    Thread T1 → Turn TB → agentMessage = "The answer is..."
```

That duplication is intentional.

GraphChat’s copy supports:

- Immediate rendering
- Search
- Google Drive sync
- Export/import
- Offline viewing
- Manual presentation edits
- Mobile viewing without a local Codex runtime

Codex’s copy supports:

- Continuing model context
- Reasoning continuity
- Tool history
- Compaction
- Forking and resuming

An important consequence: editing `Node.text` in GraphChat does not automatically rewrite Codex’s historical turn. We will need explicit behavior for “edit presentation only” versus “edit and regenerate history.”

## Why Threads Appear Under Projects

A Codex thread has a persisted `cwd`:

```text
Thread T1
cwd = /Users/sen/Code/graphchatv1
```

The desktop app appears to group threads by that path, while separately storing project ordering and saved workspace roots in its UI state.

So “GraphChatV1” in the Codex sidebar is approximately:

```text
Project UI group
├── workspace root: /Users/sen/Code/graphchatv1
├── Thread T1 with that cwd
├── Thread T2 with that cwd
└── Thread T3 with that cwd
```

The project grouping is mostly a **Codex app presentation concept**. The `cwd` attached to each thread is a **Codex Core concept**.

## Who Owns What?

| Data | Owner |
|---|---|
| Thread history | Codex Core |
| Turn and item history | Codex Core |
| Thread persistence | Codex Core |
| JSON-RPC access | App Server |
| Active thread processes | App Server + Codex Core |
| Thread `cwd` | Codex Core |
| Project grouping/order | Codex desktop app |
| Actual project files | Filesystem/user |
| Graph structure | GraphChat |
| Graph node text/layout | GraphChat |
| Graph-to-Codex references | GraphChat |

The App Server is therefore not really the database owner. It is the long-running protocol process through which clients ask Codex Core to create, load, fork, and operate on persisted threads.

## Important Detail for Our Fork

GraphChat Codex currently launches App Server using the normal Codex environment. Consequently, it uses the same default:

```text
~/.codex
```

as your installed Codex app.

That means GraphChat-created Codex threads may also be visible to the official Codex app, grouped according to their `cwd`. Likewise, both clients can resume the same underlying thread.

Our separate GraphChat application storage protects its graph and sync data, but it does **not currently isolate the Codex thread store**.

My preference is to keep the shared Codex store initially. It gives us native interoperability with the official Codex app. GraphChat should separately store its graph and its `threadId`/`turnId` mappings.
