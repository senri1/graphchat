A full rewrite would be risky and unnecessary. The current app has several solid subsystems, but its **application orchestration layer needs restructuring** before Codex is added deeply.

I’d use a selective rewrite: preserve the working UI/engine/storage code, while rebuilding the generation and state-coordination layer in the new repo.

**Keep**
These parts are already reasonably separated and valuable:

- Canvas/world engine under `src/engine/`
- React editor components under `src/components/`
- Graph node data model
- Attachment and payload storage
- Desktop filesystem persistence
- Workspace/sidebar tree
- LaTeX compilation and Synctex support
- Google Drive snapshot format, though initially disabled
- Markdown, PDF, and ink rendering
- Existing visual styling and interaction behavior

Rewriting those would consume a lot of time while introducing regressions unrelated to Codex.

**Rewrite**
The largest problem is [App.tsx](/Users/sen/Code/graphchatv1/src/App.tsx), which is over 10,000 lines and currently coordinates:

- UI state
- Graph mutations
- Persistence
- Provider selection
- Streaming
- Background requests
- Tool loops
- Cancellation
- Sync
- Import/export
- Codex-like task recovery

The current `startOpenAIGeneration` alone combines GraphChat state management with Responses API protocol details. Codex would make that substantially harder to reason about.

I’d extract or rewrite these boundaries:

```text
src/application/
  GraphChatController.ts
  GenerationController.ts
  PersistenceController.ts
  SyncController.ts

src/domain/
  graph/
  conversations/
  generation/
  attachments/

src/codex/
  CodexClient.ts
  CodexProcess.ts
  CodexThreadManager.ts
  CodexEventAdapter.ts
  CodexToolBridge.ts
  CodexTypes.ts
```

Electron would own the process:

```text
electron/codex/
  process-manager.cjs
  json-rpc-client.cjs
  ipc-handlers.cjs
```

**The Important New Model**
I would make generation a state machine rather than a large async function:

```ts
type GenerationState =
  | { status: 'idle' }
  | { status: 'starting-thread' }
  | { status: 'running'; threadId: string; turnId: string }
  | { status: 'awaiting-approval'; request: ApprovalRequest }
  | { status: 'completed'; checkpoint: CodexCheckpoint }
  | { status: 'failed'; error: string }
  | { status: 'interrupted' };
```

Codex notifications would become actions processed by the controller:

```ts
dispatch({ type: 'agent-text-delta', nodeId, delta });
dispatch({ type: 'reasoning-delta', nodeId, delta });
dispatch({ type: 'approval-requested', request });
dispatch({ type: 'turn-completed', checkpoint });
```

This would make restart recovery, cancellation, approvals, and multiple simultaneous branches much safer.

**Storage Model**
I would retain the existing portable `ChatNode` structure, but separate visual data from execution data:

```ts
type ChatNode = {
  id: string;
  parentId: string | null;
  content: NodeContent;
  layout: NodeLayout;
  generation?: GenerationRecord;
};

type GenerationRecord = {
  runtime: 'codex';
  status: GenerationStatus;
  model: string;
  checkpoint?: CodexCheckpoint;
  error?: string;
};
```

The current model mixes content, layout, API payloads, LaTeX state, generation state, and view state in one large union. We can improve that incrementally without replacing the storage format all at once.

**Approach**
I’d follow a “replace while running” approach:

1. Copy the repository into the isolated Codex repo.
2. Change app identity and storage location immediately.
3. Verify the copied app still runs.
4. Add migration tests around persisted chats.
5. Extract graph mutations and persistence from `App.tsx`.
6. Build the Codex process/client diagnostic layer.
7. Replace generation orchestration with the new Codex controller.
8. Add immutable thread checkpoints and branching.
9. Remove Responses/provider code only after Codex behavior works.
10. Continue reducing `App.tsx` without redesigning the canvas UI.

So: **do not rewrite GraphChat from scratch**. Rewrite the parts that currently make Codex integration difficult, while preserving the mature interaction, rendering, and storage work. That gives us cleaner architecture without spending months rediscovering every edge case the current app already handles.
