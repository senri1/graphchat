You’re right to push back. If we care about reasoning continuity, cache locality, and native branching, **fresh thread per generation is probably the wrong end-state**. It’s an okay prototype shape, but not the architecture I’d choose if we’re doing this properly.

**Reasoning Tokens**
In your current Responses workflow, reasoning continuity is app-managed. The model produces hidden reasoning tokens; the API can return an opaque `reasoning.encrypted_content` payload when requested. You store the raw response output and pass those reasoning items back on later turns, so the next request can use prior reasoning without exposing chain-of-thought. OpenAI’s docs describe this as the stateless/manual-context path for reasoning models. Source: [Reasoning models](https://platform.openai.com/docs/guides/reasoning?api-mode=responses), [Responses API reference](https://platform.openai.com/docs/api-reference/responses/create).

With **Codex App Server**, we should not manually shuttle `reasoning.encrypted_content` around. Codex owns the thread’s model-visible history. App Server exposes `reasoning` items and reasoning summary deltas, but the important bit is: `thread/resume` continues stored history, and `thread/fork` copies stored history into a new thread. That is the closest analogue to “keep passing encrypted reasoning items back,” except Codex is managing it. Source: [Codex App Server](https://developers.openai.com/codex/app-server).

**Token Caching**
Prompt caching is based on exact prefix reuse, not just conversation IDs. So a fresh Codex thread could still get cache hits if we inject the exact same long prefix every time. But it is more fragile.

A persistent/forked thread is better because the model-visible prefix naturally stays stable: system/developer instructions, tools, old messages, and prior reasoning history remain in the same order, and new turns append at the end. OpenAI’s prompt caching docs say cache hits require exact prefix matches, static content should go first, and cached prefixes generally last 5-10 minutes of inactivity, up to one hour for in-memory cache, or up to 24h with supported extended caching. Source: [Prompt caching](https://platform.openai.com/docs/guides/prompt-caching).

One caution: for ChatGPT/Codex subscription limits, OpenAI doesn’t publicly document the same billing semantics as API cached-token pricing. But architecturally, persistent/forked threads are still the best way to preserve reusable prefix and reasoning state.

**Yes, Use `thread/fork`**
I agree with you now: if we don’t want shortcuts, the design should use Codex threads as branch checkpoints.

The clean model is:

1. GraphChat remains the visible source of truth.
2. Each completed assistant node stores a `codexThreadId` representing the Codex history up to that node.
3. When you continue from that node, GraphChat calls `thread/fork` on that stored thread.
4. It starts the new user turn on the forked thread.
5. The new assistant node stores the new fork’s `codexThreadId`.

That mirrors GraphChat branching beautifully: every assistant node becomes a checkpoint, and every child branch gets its own Codex fork.

The extra work is handling edits. If the user edits an old node, any descendant `codexThreadId`s become stale because Codex history no longer matches GraphChat history. We’d track a `historyHash` per node and, when it mismatches, rebuild from the nearest valid ancestor or start a fresh thread and inject the branch history.

So my revised recommendation: **build the proper Codex-native version**. Use `thread/start` only for roots, `thread/fork` for branches, `turn/start` for each new user prompt, and store Codex thread metadata on GraphChat nodes. Fresh-thread-per-generation is simpler, but you’re right: it gives up too much of what Codex threads are for.
