Yes, we agree: **one GraphChat chat/graph should normally have exactly one working folder, and an individual Codex thread should never silently mix working folders.**

**Current Behavior**

Right now, GraphChat lets you change the selected folder at any time.

If you create a thread in folder A, select folder B, and continue, GraphChat currently calls:

```text
thread/resume(threadId, cwd = folder B)
turn/start(...)
```

Codex technically allows this. Its protocol permits overriding `cwd` when resuming a thread or starting a turn, and that override applies to subsequent turns.

But this is not behavior we should expose casually.

**Why Mixed Folders Are Problematic**

The App Server models a thread with one current `cwd`, not a list:

```ts
Thread {
  id: string;
  cwd: string;
}
```

It also filters threads by exact `cwd` when listing them. Therefore changing the working folder may cause:

- Earlier tool calls to refer to folder A while later calls use folder B.
- Different `AGENTS.md` and `.codex/config.toml` instructions halfway through.
- Different Git repositories and sandbox boundaries.
- The official app potentially grouping the thread under its new/current project.
- Confusing file references in the transcript.
- A thread that is difficult to interpret or resume safely.

Individual command records can retain their own working directory, but the thread itself still exposes one current project association.

The exact official-app sidebar behavior after changing a thread’s `cwd` is not documented, so we should not depend on it.

**Recommended Rule**

Each GraphChat chat should have:

```ts
type WorkingContext =
  | {
      kind: "project";
      cwd: string;
    }
  | {
      kind: "no-project";
      cwd: string; // Codex-managed working directory
    };
```

The behavior should be:

1. Before the first message, you may freely choose the working folder.
2. The first Codex thread locks the chat to that folder.
3. Every continuation and GraphChat branch inherits it.
4. Selecting another folder does not mutate existing threads.
5. To work elsewhere, create a new GraphChat chat.
6. Relocating a repository is a separate explicit recovery operation.

The folder button should become something like **Working folder**, displaying the current project path.

**GraphChat Sidebar Folders**

I do not think the arbitrary folders in GraphChat’s left sidebar should automatically represent working folders.

Those folders are currently organizational:

```text
Research
  Chat about authentication
  UI ideas

Current projects
  GraphChat work
```

A single organizational folder may reasonably contain chats for different repositories. Moving a chat between sidebar folders should not change Codex’s sandbox or project.

Better separation:

- **GraphChat folder:** user organization.
- **Working folder:** Codex execution context associated with each chat.
- Later, add filters or badges such as `graphchat-codex` to group chats by working folder.

We could eventually provide project-backed sidebar groups, but they should be an explicit type rather than changing the meaning of every existing folder.

**No Project Mode**

Your interpretation of the Codex app dropdown is close, with one detail.

When you do not select **Work in a project**, Codex still has a working directory. The official app uses a Codex-managed directory, normally:

```text
~/.codex/threads
```

It is simply not tied to one of your repositories. This mode is intended for research, planning and connected-tool work.

Therefore GraphChat should eventually support:

- **Work in project:** user-selected folder.
- **No project:** Codex-managed neutral working folder.

Currently, GraphChat requires a selected folder. If we simply omitted `cwd`, our App Server process might use its process directory, which is not a deliberate no-project implementation. We should add no-project mode properly.

**Thread Storage**

Correct: the working folder is not where the thread itself is stored.

There are three separate stores:

- **Codex state:** native threads, turns and items under Codex-managed storage.
- **Working folder:** files and commands available to the agent.
- **GraphChat storage:** graph layout, edges, transcript mirrors and Codex references.

Changing or deleting the working folder does not itself delete the Codex thread. It may make further tool usage impossible until the project is located again.

**Small Exception: Relocated Projects**

Sometimes the same repository genuinely moves:

```text
/Users/sen/Old/project
→
/Users/sen/Code/project
```

We should offer an explicit **Locate moved project** operation. It should warn the user and ideally confirm that the new folder represents the same Git repository. That is different from casually changing projects mid-conversation.

So the next correction should be to rename the control, lock the working folder after the first turn, enforce the stored `cwd` on continuations and branches, and add an explicit new-chat path for selecting another project.
