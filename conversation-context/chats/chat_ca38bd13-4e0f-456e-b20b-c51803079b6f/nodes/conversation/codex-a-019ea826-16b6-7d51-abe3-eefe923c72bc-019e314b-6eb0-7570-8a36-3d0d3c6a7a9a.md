I looked at it. My take: **OpenClaw is using the right pattern**, but it is not “swap OpenAI API key for Codex OAuth.” It routes agent turns through **Codex App Server**, and Codex App Server owns ChatGPT login, native Codex threads, streaming events, tool execution, account state, and rate limits.

Your app currently calls the normal OpenAI Responses API directly from the renderer via [src/services/openaiService.ts](/Users/sen/Code/graphchatv1/src/services/openaiService.ts). To use Codex subscription limits, GraphChat needs a **separate Codex runtime path**, probably desktop-only at first.

**How I’d Build It**
1. Add an Electron main-process Codex bridge that spawns:

```bash
codex app-server --listen stdio://
```

Then expose safe IPC methods through [electron/preload.cjs](/Users/sen/Code/graphchatv1/electron/preload.cjs): `codexAccountRead`, `codexLoginStart`, `codexRateLimitsRead`, `codexThreadStart`, `codexTurnStart`, `codexTurnInterrupt`.

2. Add “Sign in with ChatGPT for Codex” in Settings. I’d use the **device-code flow** first because it is simpler for Electron: call `account/login/start` with `type: "chatgptDeviceCode"`, show the URL/code, then wait for `account/updated`. OpenAI’s App Server docs explicitly support this and `account/rateLimits/read`.

3. Add a new provider/runtime, not just a new model string. Something like:
   - `provider: "codex"`
   - models discovered by `model/list`, or static initially: `gpt-5.5`, `gpt-5.4`, etc.
   - UI label: `Codex - GPT-5.5 via ChatGPT`

4. Add `startCodexGeneration` beside your existing `startOpenAIGeneration` in [src/App.tsx](/Users/sen/Code/graphchatv1/src/App.tsx). Convert the selected graph branch into Codex input items, start/resume/fork a Codex thread, stream `item/agentMessage/delta` into the assistant node, and persist Codex thread IDs on nodes so branching can use `thread/fork`.

5. Keep GraphChat’s direct OpenAI API path intact. Codex should be an option, not a replacement. Your current OpenAI tool loop and graph/LaTeX function tools won’t automatically transfer to Codex; later, we’d re-expose GraphChat tools through App Server dynamic tools or MCP.

The big caveat: **Codex auth only buys you Codex runtime usage**, not general OpenAI API access. Trying to extract ChatGPT/Codex tokens and send them to `/v1/responses` would be brittle and the wrong side of the line. The App Server route is the clean route.

OpenClaw’s docs confirm this pattern: it uses canonical `openai/gpt-*` models but routes through a `codex` runtime, keeps Codex auth as a profile, and even strips `OPENAI_API_KEY`/`CODEX_API_KEY` from the spawned Codex process so subscription turns don’t accidentally bill through API keys. Sources: [OpenAI Codex App Server](https://developers.openai.com/codex/app-server), [OpenClaw Codex harness](https://documentation.openclaw.ai/plugins/codex-harness).
