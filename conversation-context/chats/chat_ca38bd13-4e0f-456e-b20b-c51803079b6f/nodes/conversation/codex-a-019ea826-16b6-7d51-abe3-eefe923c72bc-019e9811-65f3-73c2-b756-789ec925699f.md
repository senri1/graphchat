**Thread Mapping**
There is one important correction to make explicit: `thread/fork` forks the current stored tip; it does not accept an arbitrary historical turn.

Therefore, every completed assistant node should represent an **immutable Codex checkpoint**:

```text
Assistant A -> thread T1

Continue from A:
    fork T1 -> T2
    run next turn on T2

Create another branch from A:
    fork T1 -> T3
    run alternate turn on T3
```

We never append another turn directly to `T1`. This means:

- Every assistant node has its own Codex thread ID.
- Parent checkpoints remain unchanged.
- Sibling branches cannot leak into one another.
- Reasoning history and model-visible context are copied natively by Codex.
- The Codex `sessionId` connects the resulting thread tree.

This creates more local Codex thread logs, but it is the cleanest correctness-first mapping.

**Node Metadata**
I would add:

```ts
codexCheckpoint?: {

  sessionId: string;

  agentMessageItemId?: string;

  effort?: string;

```

GraphChat still stores the visible text, layout, attachments, and parent relationship. The Codex checkpoint supplies reasoning continuity and agent state.

**Tools**
There are two categories:

- Codex built-in tools: shell, filesystem changes, web search, MCP, skills, and so forth. App Server executes these and asks GraphChat for approvals when required.
- GraphChat tools: graph-node and LaTeX operations. I would expose these through App Server’s `dynamicTools`, with GraphChat executing each `item/tool/call` request and returning the result.

Dynamic tools are currently experimental, so I would pin and test against a specific Codex version. If they prove unreliable, the durable replacement is a small GraphChat MCP server.

**Working Directory**
Each GraphChat chat gets an optional Codex project folder:

```ts
codexWorkspace: {
  cwd: string;
  sandbox: 'readOnly' | 'workspaceWrite';
  approvalPolicy: string;
};
```

That folder is supplied to `thread/start` and determines where Codex’s file and command tools operate.

Browser/mobile Codex generation would remain unavailable initially because they cannot spawn App Server.
